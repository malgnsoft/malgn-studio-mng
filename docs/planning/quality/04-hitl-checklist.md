# 04 — HITL 검수 체크리스트

> 범위: 검수자(SME/Reviewer)용 점검 체크리스트(공통+산출물별), 반려 사유 분류, 승인/반려 흐름(05-GENERATION 상태머신 연계), 4-eyes(자가검수 금지) 원칙.
> 정본 정합: 05-GENERATION §5(HITL 게이트·상태모델·검수자 역할). 루브릭은 `03`, 평가 정합은 `02`.
> 사용처: 검수자가 직접 체크. frontend가 검수 UI에 이 체크리스트·반려 사유를 구현.

---

## 1. 4-eyes(자가검수 금지) 원칙

- **원칙**: 산출물의 **작성자(Author)와 검수자(Reviewer)는 동일인이 아니어야 한다**(4-eyes). 작성자가 자기 산출물을 스스로 승인하지 못한다.
- **MVP 예외**(05-GENERATION §5.2): MVP에서는 작성자=검수자 1인 체제도 **허용**하되, **검토 단계는 강제**한다(셀프 리뷰라도 명시적 승인 액션 필요). 즉 1인이라도 draft→in_review→approved의 명시적 전이를 거친다. 1인 자가승인은 **임시 운용**이며, 신뢰가 중요한 B2B/공공 발행물은 별도 검수자 배정을 권장(가정).
- **하드 규칙**: 자동 승인 없음 — published는 반드시 approved를 거친다(05-GENERATION §5.1).

---

## 2. 승인/반려 흐름 (상태머신 연계)

> 05-GENERATION §5.1 상태모델: `generating → draft → in_review → (approved | draft) → published`

```
draft ──[검토 요청]──► in_review ──┬─[승인]─► approved ──[발행]─► published
                                   └─[반려(사유 필수)]─► draft (재생성/수정)
```

| 단계 | 검수자 액션 | 전제 조건 |
|---|---|---|
| in_review 진입 | 검수 시작 | 작성자가 검토 요청, source_refs 누락 0(하드) |
| 승인(approve) | 체크리스트 전 항목 통과 + 하드 게이트 충족 | 4-eyes(작성자≠검수자, MVP 예외 §1) |
| 반려(reject) | **반려 사유 분류 필수**(§4) + 코멘트 | 부분 코멘트는 `target_path`로 위치 지정(예: `items[3]`) |
| 발행(publish) | approved 상태에서만 | 출처 게이트·정확성 게이트 통과 |

- **출처 게이트**: source_refs 누락 산출 단위가 있으면 **승인 불가**(05-GENERATION §5.1, 하드).
- 반려는 `review.decision = rejected` + `comment` + 사유 분류로 기록(05-GENERATION §6.6).

## 3. 공통 검수 체크리스트 (8종 전체 선적용)

> 루브릭 `03` §2의 5축을 검수 행위로 변환. 모든 산출물에 적용.

- [ ] **C1 정확성** — 사실·수치·개념 오류·환각이 없는가? (하드: 오류 발견 시 반려)
- [ ] **C2 그라운딩** — 모든 산출 단위에 source_refs가 있고, 출처가 실제 내용을 지지하는가? (하드: 누락 시 승인 불가)
- [ ] **C3 교육적 구조** — 학습목표(`objective_ref`)와 정렬되고 논리적 계열·인지부하가 적정한가?
- [ ] **C4 난이도 적합** — 대상·블룸·난이도가 적절한가?
- [ ] **C5 접근성·편향** — 대체텍스트/자막 가능, 차별·편향 표현이 없는가? (차별·편향은 하드)
- [ ] **앵커 무결성** — 모든 산출물이 유효한 학습목표를 참조하는가? (앵커 없는 파생물 불허)

## 4. 반려 사유 분류

반려 시 아래 7개 분류 중 ≥1개를 선택한다(복수 가능). 분류는 재생성 파라미터·통계·QA에 활용된다.

| 코드 | 분류 | 정의 | 연관 게이트 |
|---|---|---|---|
| `R1_FACT` | 사실 오류 | 잘못된 사실·수치·개념(환각 포함) | C1 하드 |
| `R2_SOURCE` | 출처 누락/부정확 | source_refs 누락 또는 출처-내용 불일치 | C2 하드 |
| `R3_OBJECTIVE` | 목표 불일치 | 학습목표 미정렬·미커버, 앵커 누락 | C3·`02` |
| `R4_DIFFICULTY` | 난이도 부적합 | 블룸·난이도 부적절, 블룸 미달 문항 | C4·`02` §3 |
| `R5_EXPRESSION` | 표현 문제 | 문장 오류·모호·중복·산만 | C3 |
| `R6_ACCESS` | 접근성 결함 | 대체텍스트·자막·색의존 등 | C5 |
| `R7_BIAS` | 편향/포용성 | 차별·고정관념·다양성 결여 | C5 하드 |

> Critical(하드 차단): R1·R2·R7. 발견 시 승인 불가. R3~R6은 검수자 판단(경미 시 코멘트 후 승인 가능, `03` §1.2).

## 6. 검수자 역할·책임 (참조)

> 05-GENERATION §5.2.

| 역할 | 권한 | 책임 |
|---|---|---|
| 작성자(Author) | 생성·편집·재생성·검토요청 | 초안 품질, 출처 확인 |
| 검수자(Reviewer/SME) | 승인·반려·코멘트 | 내용 정확성, 교수설계 정합, 출처 타당성 |
| 발행자(Publisher) | 발행·개정 | 발행 시점·버전 관리 |
| 관리자(Admin) | 정책·게이트 설정 | 워크플로 거버넌스 |

- 교수설계 정합(목표 커버·블룸·난이도) 판정은 **검수자(SME)**의 책임 영역이다. 본 체크리스트가 그 판단 기준을 제공한다.

---

## 7. 부분 재생성·재검수

- 반려 후 부분 재생성(슬라이드 1장·문항 1개 단위, 05-GENERATION §5.3)이 가능하다. 재생성된 단위는 다시 draft → in_review를 거친다(부분이라도 검수 강제).
- 재검수 시 변경된 단위만 집중 점검하되, 변경이 상위 앵커(목표·커리큘럼)에 영향 주면 하위 stale 여부를 재확인한다(`01` §5.2).
