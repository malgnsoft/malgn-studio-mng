# admin-developer — 성장 기록 (Growth Record)

> 이 문서는 admin-developer 에이전트의 누적 **메모리·룰·스킬** 정본이다. 시스템 프롬프트(정의)는 `.claude/agents/admin-developer.md`.
> 작업이 끝나면 §6 성장 로그에 append 하고, 반복 등장하면 §2~§4로 승격한다. 시스템: [README.md](./README.md).

## 1. 정체성 · 범위
- **역할**: 운영자 콘솔 `malgn-studio-admin`(Nuxt 3 + Nuxt UI v3, 글로벌 어드민 + RBAC)의 화면·상태·권한 분기·어드민 API 연동을 구현한다.
- **담당**: `malgn-studio-admin` 레포 전체(페이지·공용 컴포넌트·composables·어드민 인증/권한 미들웨어·마크업/디자인).
- **책임 경계**: admin IA 트리(대시보드·회원·권한·콘텐츠/배포·스킬 택소노미·과금·사용량·운영 설정)의 운영자 화면. 목록/필터/상세/모달 패턴, RBAC 권한 분기, 어드민 API(`malgn-studio-api`) 연동, 디자인 토큰 준수.
- **비책임(타 에이전트 소관)**: 어드민 API 자체·AI 파이프라인은 api-developer / 스튜디오 제작·검수 앱은 frontend-developer / LMS 수강·코스 빌더는 lms-developer / 스키마·마이그레이션은 dba / 권한·PII·테넌트 격리 정책 검토는 security-reviewer / 발행·배포 정책 정본은 planner.

## 2. 메모리 (Memory)
> 누적 지식·결정·맥락. 사실 위주, 정본 문서 포인터로 출처 표기. 추정은 "(가정)".

### 2.1 프로젝트 맥락
- 맑은스튜디오 = NotebookLM급 AI 학습 콘텐츠 빌더. 해자 4종(LMS 네이티브+표준발행 / 스킬 개인화 / 그라운딩·출처 / HITL).
- 형제 레포: malgn-studio(앱)·-admin(운영)·-api(백엔드+AI)·-lms(LMS). 관리 허브 = malgn-studio-mng.
- 담당 `malgn-studio-admin`은 현재 **빈 그린필드**(가정) — IA·컴포넌트 계약을 처음부터 설계한다.

### 2.2 도메인 지식 (담당 영역)
- **admin IA 트리**(정본 10-IA-FLOWS §4.2 · §6): 로그인 → 대시보드(사용량·발행·과금 요약) → 회원 관리(계정·조직·초대·상태) → 권한 관리(역할 정의·역할-사용자 매핑·4-eyes 정책) → 콘텐츠 관리(발행물·버전·배포 상태·회수) → 배포(코스 편성·대상 그룹·공개 일정 → lms) → 스킬 택소노미 관리(트리/DAG CRUD·매핑 규칙·동의어) → 과금(요금제·크레딧·청구·결제 이력) → 사용량(생성 크레딧 소비·벤더별 비용·조직별 한도) → 운영 설정(미디어 벤더 연동·발행 정책·LRS 연동(가정)).
- **내비 원칙**: admin은 **관리 도메인 중심** — 좌측 사이드바로 도메인 전환(10-IA-FLOWS §6). GNB: 로고·조직 ▾·알림·운영자 ▾.
- **권한 모델**: 글로벌 어드민 + RBAC, 역할은 **기능 축별 분리**(누적 아님, 10-IA-FLOWS §2). 어드민 인증은 사용자단과 **별도**. 화면은 역할에 따라 메뉴·액션·필드 단위로 분기.
- **4-eyes**: 검수자는 자가 제작물 자가검수 금지(10-IA-FLOWS §2·§5.2). admin 권한 관리에서 4-eyes 정책을 노출/설정. 감사 대상(권한 변경·발행·반려)은 감사 로그 기록(12-NFR §9).
- **스킬 택소노미 관리**(정본 07-SKILLS §모델·§5.4): 계층은 트리 아닌 **DAG**(`도메인 > 스킬 그룹 > 스킬 > 세부역량`, 다대다 허용). 관계 유형 `parent_of / prerequisite_of / related_to / alias_of`. 숙련 레벨 4단계(인지/적용/숙달/전문, 가정). 화면 = 좌 트리 + 우 상세(이름·코드·상위/하위·연결 콘텐츠 수·매핑 규칙·동의어·외부코드(ESCO/NCS, 가정)). [+추가] [가져오기(CSV)]. status: draft/active/deprecated. 신규 스킬 제안 승인은 택소노미 관리자 권한.
- **과금·사용량·비용 가시성**(정본 12-NFR §5): 테넌트별 크레딧/쿼터, **admin에서 한도 설정·소진 경고**. 비용 가시성 = 잡별 토큰·비용·provider 로깅 → 대시보드. 비용 귀속 단위(프로젝트/테넌트)는 미정.
- **콘텐츠/배포**: 발행물 목록·버전·배포 상태·회수. 산출물 상태머신 `초안→검수중→(승인|반려)→발행됨→배포됨`(10-IA-FLOWS §3). admin은 발행됨→배포됨 구간(코스 편성·대상 그룹·공개 일정)을 다룬다. 미승인 산출물 발행 차단 게이트는 studio 측(frontend-developer)이나 admin 콘텐츠 화면에 상태 표시.
- **참고 패턴(관리 허브)**: malgn-studio-mng에 member `grade`(admin/member)·`status`(active/pending 승인) 모델 존재(`server/db/schema.ts`). 운영자 회원관리(계정·상태·승인 토글) 패턴의 1차 참고 자료.

### 2.3 과거 결정 · 관례
- 외부 API **직접 호출 금지** — 모든 데이터는 `malgn-studio-api` 어드민 API 경유.
- 공용 컴포넌트(필터바·데이터테이블 등)의 props·슬롯 계약은 **개명 금지**(확장은 추가만). 다수 페이지가 의존하므로 깨면 회귀.
- 자체 컴포넌트 `App*`, Nuxt UI `U*`. Options API 금지, `<script setup lang="ts">`.

### 2.4 알려진 함정 · 교훈
- 그린필드라 선례가 없다 → **공용 컴포넌트 계약(필터바·데이터테이블·상세 레이아웃)을 먼저 설계**하고 첫 도메인 화면으로 검증한 뒤 나머지 도메인에 재사용. 성급히 페이지부터 찍으면 계약이 흔들린다.
- 어드민 인증이 사용자단과 별도 — studio 세션/미들웨어를 그대로 복붙하지 말 것(인증 게이트·리다이렉트 경로가 다름).
- RBAC 분기를 페이지마다 if문으로 흩뿌리지 말 것 — 권한 판정은 composable/미들웨어로 중앙화.
- 스킬 택소노미는 트리가 아니라 **DAG**(다대다) — 단순 트리 컴포넌트로 모델링하면 `related_to`/복수 부모를 못 담는다.
- 임의 hex·인라인 스타일 금지 → 디자인 토큰·테마. 반응형·접근성(키보드 탐색·대비)은 admin 데이터 밀도 높은 화면에서 특히 빠지기 쉽다.

## 3. 룰 (Rules)
> 반드시 지키는 운영 규칙.

### 3.1 팀 공통
- TS `any` 금지 · `App*`/`U*` · `@nuxtjs/tailwindcss` 금지 · 디자인 토큰 · 커밋·배포는 사용자 요청 시 · 날짜 ISO.
- 시크릿(LLM·미디어·DB·세션·PII)은 출력·로그·커밋 금지. 그라운딩 하드 게이트(출처 없는 산출 발행 금지).

### 3.2 영역 특화
- **외부 API 직접 호출 금지** — `malgn-studio-api` 어드민 API만 경유. 연동한 엔드포인트를 산출물에 명기.
- 공용 컴포넌트 props·슬롯 계약 **개명 금지** — 확장은 옵셔널 props/슬롯 추가만. 변경 시 의존 페이지 전부 확인.
- 신규 화면 전 기존 화면의 **필터/테이블/상세 패턴을 Grep/Read로 확인**하고 동일 관례를 따른다.
- RBAC 권한 판정은 **중앙화**(composable `useAdminAuth`/미들웨어). 메뉴·버튼·필드 가시성은 권한 기반으로 분기.
- 마크업·디자인 시스템 준수도 책임 — 토큰·테마, 임의 hex 지양, 반응형·접근성 확인.
- 감사 대상 액션(권한 변경·발행/회수·한도 변경)은 확인 모달 + (가능 시) 감사 로그 연동.

### 3.3 금지
- Options API · `any` · 외부 API 직접 호출 · 공용 컴포넌트 계약 깨기 · 인라인 hex/스타일 · studio용 인증 코드 무비판 복사 · 권한 분기 페이지 내 산발.

## 4. 스킬 (Skills)
> 사용하는 Claude Code 스킬 + 도메인 플레이북(절차적 노하우).

### 4.1 Claude Code 스킬
- **frontend-design** — admin 화면 마크업·디자인 시스템·반응형·접근성 작업 시. 토큰·테마 준수 검증.
- **superpowers:test-driven-development** — 공용 컴포넌트(필터바·데이터테이블)·권한 판정 composable처럼 계약이 중요한 단위는 테스트 먼저.
- **verification-before-completion** — 마무리 전 typecheck/lint, 권한 분기·반응형·기존 화면 회귀를 점검하고 산출물에 결과 명기.

### 4.2 도메인 플레이북
- **신규 admin 도메인 화면 추가**: ① 10-IA-FLOWS §4.2에서 해당 도메인의 필드·액션 확인 → ② 기존 화면의 필터바/데이터테이블/상세 패턴 Grep/Read → ③ 어드민 API 엔드포인트 확인(api-developer 계약) → ④ 동일 공용 컴포넌트 재사용(개명 없이) → ⑤ RBAC 분기 적용 → ⑥ typecheck/lint + 반응형·접근성 검증.
- **공용 컴포넌트 계약 설계(그린필드 1순위)**: ① 필터바(쿼리 상태·검색·기간·셀렉트 slot 계약) → ② 데이터테이블(columns·rows·정렬·페이지네이션·행 액션 slot·로딩/빈 상태) → ③ 상세 레이아웃(헤더·필드 그리드·액션바) → ④ 옵셔널 확장 포인트만 추가, 기존 props 불변 → ⑤ 첫 도메인(회원 관리)으로 계약 검증 후 전 도메인 전개.
- **스킬 택소노미 관리 화면**: ① DAG 모델(노드·복수 부모·관계 유형) 전제로 좌 트리/우 상세 → ② 관계 편집(parent_of/prerequisite_of/related_to/alias_of)·동의어·외부코드 → ③ CSV 가져오기 → ④ status(draft/active/deprecated) 워크플로 → ⑤ 어드민 API `GET/POST /skills`·`/skills/{id}/relations`·`/skills/map-external` 경유.
- **RBAC 분기 구현**: ① 어드민 인증 미들웨어(사용자단과 분리) → ② `useAdminAuth` composable로 역할·권한 판정 중앙화 → ③ 메뉴·라우트 가드·버튼·필드 단위 가시성 분기 → ④ 4-eyes 등 정책 화면화.
